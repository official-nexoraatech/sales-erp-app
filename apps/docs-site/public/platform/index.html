Developer Platform Overview

Eleventh and final module documented from a direct code audit. Unlike every prior module, this one has no client or admin audience — it's architecture conventions and "how do I add X" guidance for engineers working in this codebase, synthesized from patterns confirmed across all ten business modules already documented here.

This monorepo is a pnpm + Turborepo workspace: 18 apps (15 backend services + 2 frontends + this docs site) and 8 shared packages. If you've read any of the other ten modules, you've already seen this platform's two defining, repeating traits in action:

  • The permission-drift pattern — a role's granted permissions not matching what a route actually checks — appeared in nearly every business module audited. It has a name and a paper trail in this codebase (see Adding a Page, Permission, or Endpoint), and there's still no automated check preventing it from recurring.
  • The event-payload-mismatch pattern — a service publishing one field name, a consumer expecting another — caused several of the most severe bugs found in this audit (notably in Accounting & GST). It has a root cause: no shared, typed event-payload contracts exist anywhere in the codebase. See Adding an Event Consumer Safely for the concrete fix.

Adding a Page, Permission, or Endpoint

Adding a page to web-frontend

Register a lazy-loaded route in App.tsx wrapped in the shared PermissionRoute guard, and add a matching entry to navigation.ts whose own doc-comment says its permission field must match the route guard.

One confirmed exception to this pattern

The account-level Security Settings page is deliberately registered without a permission guard — every logged-in user can reach their own security settings, which makes sense for that one page. It's worth knowing this exception exists rather than assuming the guard is truly universal when you go looking for a reference example.

pos-frontend does not use this pattern at all

POS has no router and no navigation config — it's a flat set of top-level screens shown/hidden directly in code. Don't look for the App.tsx/navigation.ts convention there; it's a structurally different app, not a variant of the same one.

Adding a permission

Add a constant to the shared permissions file, then grant it to the relevant role(s) in the tenant-service's role-defaults seed file.

This is the exact step where the recurring role/permission mismatch bug gets introduced

Nearly every business module audited on this site found at least one case of a role missing a permission its own screens or training material assume it has — or the reverse, a permission nobody's role actually needs. There's a documented paper trail of a past cleanup pass that already removed several fully-dead permission constants, but the underlying gap — nothing automatically checks that a new permission actually gets granted to the right role, or that an old one is still referenced anywhere — remains open today. If you add a permission, manually verify which role(s) should have it and add it to their seed entry in the same change, and check whether it's actually referenced by an existing route before assuming a new one is needed.

Adding a REST endpoint

A Fastify route file under a service's src/api/ directory, with an [authenticate, requirePermission(PERMISSIONS.X)] preHandler on the route — this exact pattern is consistent across every service audited on this site.

Adding a Table & Writing Migrations

Add a table definition to the relevant file under the shared DB package's schema directory, export it through the existing schema barrel, then generate a migration with the package's own generate script.

Every table needs these columns

This project's own architecture spec mandates tenant_id (always first, not null), created_at/updated_at, created_by, and a version column for optimistic locking on every table — and in practice, this is followed consistently across the schema. Tenant isolation in this system is enforced entirely at the application layer (there's no database-level row-security policy doing it for you), so a missing or forgotten tenant_id filter is a real, exploitable data-leak risk, not just a style violation.

This rule lives in a planning document, not next to the migrations folder itself

The mandatory-columns rule is real and documented, but it's in a separate architecture-spec document, not a README inside the migrations directory itself. A new engineer opening that folder cold wouldn't discover the rule without already knowing where to look.

The migration tool's own bookkeeping is currently out of sync — be careful generating a new migration right now

Every migration file that's ever been created is correctly tracked in the tool's own index — that part is fine. But the tool also needs a "snapshot" of the database schema as it stood after each migration, to correctly compute the next one, and only the very first two migrations in this project's history have a snapshot saved. Every migration added since is missing one. In practice, this means the tool's next auto-generated migration has to guess at a schema baseline that's dozens of migrations out of date — it could either fail outright or quietly try to redo something that's already been done. Before trusting an auto-generated migration today, generate it in a throwaway branch first and read the SQL it produces carefully rather than assuming it's correct.

Adding a Report

Add a definition to the report registry (slug, name, category, parameters, columns, required permission) in report-service, and a matching implementation case in the report engine right next to it.

This is the one place in the whole codebase with genuinely clean parity

Every one of the 77 currently-registered reports has exactly one matching implementation, with zero orphans in either direction — confirmed by direct comparison, twice, across two separate audits. Every other "registry vs. implementation" or "permission vs. route" pairing checked across this whole documentation project had at least one mismatch somewhere. Whatever discipline keeps these two files in sync, it's worth understanding and replicating elsewhere in the codebase.

Remember to grant the report's required permission to whichever role should actually be able to run it — the Reports module's own audit found several reports gated on permissions no real operational role holds by default, purely because that step was missed.

Adding an Event Consumer Safely

Services publish domain events to an outbox table inside their own database transaction; other services run background consumers keyed by event type to react asynchronously (posting a journal entry, updating a GST register, sending a notification, and so on).

This exact mechanism caused several of the most serious bugs found across this whole documentation project

The shared event envelope only types its outer shape — the actual business payload inside it is untyped on purpose at the platform level. Every consumer written so far responds to this by declaring its own private copy of what it expects the payload to look like, then forcibly casting the untyped data into that shape without any real validation. When a publisher's real payload doesn't match a consumer's guessed shape — a missing field, a different field name — nothing catches it at compile time or runtime. This is precisely how the accounting module ended up silently failing to post GST for ordinary domestic sales, and how sale-return accounting entries ended up posting at zero: two of the most significant defects found in this entire audit trace back to this one root cause.

The safest way to add a new one

Define the event's payload shape once, as a proper named type living alongside the event-type constant itself in the shared types package — not as a private interface re-declared inside the consumer. Have the publishing service build its event payload against that same shared type, and have the consumer import and use it directly instead of casting an untyped object. This isn't how any existing consumer in this codebase actually works today — it's a recommendation based on the fact that nothing currently stops this exact class of bug from recurring, not a documented house style you're expected to already know.

Testing & CI

Vitest everywhere. Integration tests connect to a genuinely running Postgres instance via a real database URL — there's no mocking or ephemeral test-container layer for these; if that URL isn't set, the tests silently skip rather than fail, so a green run locally without a database configured doesn't actually prove anything about the integration suite.

CI pipeline

Lint, type-check, a coverage-gated test job (≥80%, against real Postgres/Redis service containers), a mocked-boundary Playwright smoke suite for both frontends, a 14-service + gateway Docker build matrix, dependency audit, static analysis (Semgrep), container scanning (Trivy), secret scanning across full git history (TruffleHog), Snyk, a manual-dispatch load-test job, and a tag-triggered staging deploy.

Adding a new backend service to CI is entirely manual, in four separate places

There's no automatic discovery of new services anywhere in the pipeline. A new service has to be added by hand to the Docker build matrix, the container-scan matrix, and two separate hardcoded service-name lists inside the staging-deploy job's rollout and rollback steps. Miss one of the four and that service silently doesn't get its intended coverage (built but never security-scanned, or deployed but never included in an automatic rollback) with no error telling you it was left out.

Environment Setup & Bootstrapping

How to bring up a new environment — a fresh, empty database for local dev, QA, staging, or the first production deploy — from nothing to a working first tenant. This is distinct from the go-live runbook (docs/go-live-runbook.md), which migrates an existing business's data into a tenant that's already been provisioned; this page covers what happens before that tenant exists at all.

Step 1 — Run migrations

Apply every migration under the shared DB package's migrations directory, in order. 0020_es21_platform_operator.sql is required — it seeds a reserved platform-operations tenant and a PLATFORM_OPERATOR role holding exactly one permission, PLATFORM_TENANT_MANAGE, but deliberately seeds zero users.

Step 2 — Bootstrap the first PLATFORM_OPERATOR (once per environment)

DATABASE_URL="<env-db-url>" pnpm --filter @erp/tenant-service bootstrap-operator -- --email=<ops-email> --password=<strong-password>

Refuses to run again once a PLATFORM_OPERATOR user already exists, so it's safe to run exactly once right after migrations. Log in via the normal /auth/login with that email/password and whatever tenantId the platform-operations tenant actually got in that environment — do not assume it's 1; confirm with SELECT id FROM tenants WHERE slug='platform-operations'. This account can only manage tenant lifecycle; it has no sales/inventory/accounting access of its own.

apps/tenant-service/scripts/bootstrap-platform-operator.ts

Step 3 — Provision the environment's first real tenant

Using the operator's token, call POST /admin/tenants with name, slug, contactEmail, adminFirstName/adminLastName, adminPassword, and plan. This creates the tenant, seeds its roles, and creates one OWNER-role user from contactEmail/adminPassword — that OWNER account, not the platform operator, is what real users work with day to day.

apps/tenant-service/src/domain/TenantProvisioner.ts

Step 4 — Seed auth notification templates, per tenant

Call POST /api/v2/notifications/templates/seed-auth with { "tenantId": <new-id> } once for every newly provisioned tenant, including platform-operations itself (which has never had this run — see the recovery script below for why that matters). Skipping it means that tenant's forgot-password and other auth emails silently no-op.

Security model — who can actually run the bootstrap/reset scripts

Not gated by the app at all — DB credential + shell access is the entire trust boundary

Both the bootstrap script above and the password-reset script below connect straight to Postgres via DATABASE_URL. There is no HTTP route, no RBAC check, no application-level auth wrapping either one. Anyone who holds an environment's DATABASE_URL and has shell access to run pnpm/tsx against it (a CI runner, a bastion host, a pod you can kubectl exec into) can run these — equivalent to DBA-level trust, not a scoped platform-operator permission. This is intentional: they exist for the case where no operator account exists yet, or the only one is locked out. In production, DATABASE_URL is already Vault-gated (see docs/vault-rollout.md) — apply the same discipline to QA/staging credentials, don't leave them in a shared dev .env. Neither script writes an audit-log entry, so the only record of who ran one and when lives in your infra logs (CI job history, shell history, DB connection logs), not in the app.

Recovering a locked-out or forgotten operator password

DATABASE_URL="<env-db-url>" pnpm --filter @erp/tenant-service reset-operator-password -- --email=<ops-email> --new-password=<new-password>

Scoped strictly to the platform-operations tenant's own user row, so it can't accidentally reset an unrelated user in a different tenant that happens to share the same email address — this has actually happened in this codebase's own test data. Same security model as Step 2 applies, and it force-revokes that account's existing sessions.

apps/tenant-service/scripts/reset-platform-operator-password.ts

Known Issues & Needs Review

This is the final page of the final module in this documentation project. Ranked by impact to a developer working in this codebase.

Migration tooling's schema-snapshot baseline is 49 migrations out of date

Every migration is correctly indexed, but the schema snapshots needed to safely auto-generate the next one only exist for the first two migrations in the project's history — everything since has no corresponding snapshot. The next auto-generated migration has no reliable baseline to diff against. Confirmed as a real, currently-open state, not a stale or exaggerated concern — verify the exact file counts yourself before generating a migration if this matters to your change.

No shared, typed event-payload contracts exist anywhere

This is the single root cause behind several of the most serious defects found across this entire documentation project. Every consumer re-declares its own guess at a payload shape and force-casts into it with no real validation. See Adding an Event Consumer Safely for the concrete, actionable fix — it isn't a documented convention today, it's a gap.

The permission-drift pattern has no automated guard

A past cleanup pass already found and removed several fully-dead permission constants, and the project's own architecture notes explicitly acknowledge there's still no automated check preventing new drift from recurring. This single gap is the direct cause of a role/permission mismatch found in nearly every business module documented on this site.

Local-dev bootstrap sequence is now documented, but still fully manual

The Docker compose file still only brings up infrastructure (database, message queue, object storage, search, observability stack) — none of the actual application services or frontends run inside it, and there's no single script that ties starting services, running migrations, and creating the first tenant together. What's changed: the manual sequence itself — migrate, bootstrap the first PLATFORM_OPERATOR, provision the first tenant — is now written down in one place. See Environment Setup & Bootstrapping. This matches a related gap already found independently in the Super Admin / Tenants module.

No root README, CONTRIBUTING guide, or architecture-decision-record directory

Real architecture documentation exists and is candid about known issues — including this exact permission-drift pattern — but it lives in a planning-documents folder that isn't linked from anywhere a new engineer would naturally start looking, and there's no top-level onboarding document at all.

A stale environment-variable example and an unused API gateway

The example environment file lists the wrong default port for notification-service (it actually documents purchase-service's port under the wrong name). Separately, a real, working API gateway exists in the codebase but the main web application never actually routes through it — every service is called directly instead, a fact already noted independently across several business-module audits on this site.

What's genuinely solid

The mandatory multi-tenant column convention (tenant ID, audit columns, optimistic-lock version) is real, documented, and followed consistently across the schema. The CI pipeline itself is thorough — coverage gating, static analysis, container scanning, and secret scanning are all real and wired in, not decorative. And the report registry — mentioned throughout this project as the one place with zero drift between what's declared and what's implemented — remains a genuine model for how the rest of the codebase's "two things that must stay in sync" pairs could work.

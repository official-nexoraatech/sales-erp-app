Client documentation

Inventory Overview

Third module documented from a direct code audit. Covers stock levels, warehouses, transfers between warehouses, stock adjustments, physical verification, and the stock valuation report.

This is where stock actually lives across your warehouses — separate from POS and Sales & Invoicing, which both deduct stock as a side effect of selling, not as their main job.

What's here

  • Stock Levels — current quantity per item per warehouse, with a low-stock flag.
  • Warehouses — the physical locations stock is tracked against.
  • Stock Transfers — moving stock from one warehouse to another.
  • Stock Adjustments — correcting quantity for damage, theft, shortage, and similar.
  • Physical Verification — counting actual stock and reconciling variances.
  • Stock Valuation — the ₹ value of what you're holding, by costing method.
No reservation screen

The system has a real "hold this stock for an order" mechanism internally, but nothing in the app — not POS, not Sales, not this module — currently uses it to reserve stock ahead of a sale. You won't find a reservations screen anywhere; it isn't missing from the UI by accident, it's simply not wired into any workflow yet.

Stock Levels & Warehouses

Viewing stock

The Stock Levels screen shows available quantity, reserved quantity, and your configured reorder level per item and warehouse, filterable by warehouse and by a "below reorder level" checkbox. Anything at or under its reorder level is flagged.

Can lag briefly after a sale or purchase receipt

This screen reads from a separate summary table that's kept up to date by inventory's own background processes, not instantly by the sale or purchase itself. In normal operation it catches up quickly; if a number here looks off right after a large sale or a goods receipt, give it a moment or see Stock Accuracy & Reconciliation.

Warehouses

Create and manage warehouses under Settings → Warehouses: Name, Code, which Branch it belongs to, and whether it's the branch's default warehouse.

No address fields in this screen

The system can store a full address (street, city, state, pincode) for a warehouse, but the create/edit form here doesn't expose those fields — there's currently no way to enter a warehouse's address from the UI.

Stock Transfers

How a transfer moves

1
Draft — created, nothing moved yet
→
2
Submitted → Approved
→
3
Dispatched — stock leaves the source warehouse now
→
4
Received — stock arrives at the destination

Stock is deducted from the source warehouse the moment it's dispatched — not when it's received. Between dispatch and receipt, that stock is genuinely in transit: no longer available at the source, not yet available at the destination.

A new transfer can get stuck in Draft

Creating a transfer via "New Transfer" saves it as a Draft, but there is currently no button anywhere to move a Draft transfer to Submitted. If your transfer isn't showing up for approval, this is almost certainly why — ask your developer/vendor to move it forward for you until this is fixed, or check whether a newer version has resolved it.

Cancelling after dispatch doesn't put stock back

A transfer can still be cancelled after it's been dispatched — but doing so does not restore the stock that already left the source warehouse. Avoid cancelling a transfer once it shows Dispatched; if one needs to be undone at that stage, treat it as a stock adjustment instead until this is fixed.

The transfer detail page is minimal

Opening a transfer from the list shows only its number, status, and warehouses — no line items and no action buttons. Approve/Dispatch/Receive are only available from the "⋯" menu on the Stock Transfers list, not from inside the transfer itself.

Adjustments & Physical Verification

Stock Adjustments

Use this to correct quantity for a reason other than a sale, purchase, or transfer: Damage, Expiry, Theft, Shortage, Excess, Quality Issue, Sample Issued, or Return to Vendor.

Large adjustments need approval

An adjustment worth more than ₹50,000 requires a separate approval step before it takes effect — it isn't applied the moment it's submitted.

No way to cancel a submitted adjustment

Once submitted, there's no Cancel option visible in the list — only Submit (while still a draft) and Approve (once submitted). If you created one in error, you'll need an approver to reject it through whatever process your team has agreed on, since there's no self-service cancel button today.

Physical Verification

Count actual stock in a warehouse, and the system shows the variance against what it expected — count, review variances, then approve.

Stock Valuation Report

Shows the ₹ value of stock on hand, per item and warehouse, using whichever costing method that item is set to: WACC (weighted average cost) or FIFO (first-in-first-out). The costing method is set per item, not per tenant — different items can use different methods side by side.

The "Estimated" badge

When there's no recorded stock movement yet for an item in a specific warehouse, the report falls back to a tenant-wide estimate and marks that row "Estimated" rather than showing a precise per-warehouse figure. This is expected behavior for a warehouse an item hasn't moved through yet, not an error.

The "As of Date" picker doesn't change the numbers

You can pick a date on this report, but the underlying figures are running totals, not a historical snapshot by date — changing the date currently has no effect on what's shown. Treat this report as always showing today's value, regardless of the date selected.

Access & Roles

The Inventory Manager role cannot actually manage inventory

Every write action in this module — creating or approving a stock transfer, creating or approving a stock adjustment, running physical verification, and creating/editing/deleting a warehouse — is gated by a single permission the code checks directly. The seeded Inventory Manager role is not granted that permission; it only has view-level and reservation-level access. In practice, today, only Owner and Admin accounts can perform any of these actions — an Inventory Manager account will get "permission denied" on all of them. If you've assigned this role expecting it to run day-to-day inventory operations, it currently can't, and needs a role-configuration fix (or a temporary Admin-level account) until resolved.

Stock Accuracy & Reconciliation

How stock stays correct across modules

POS and Sales & Invoicing don't call this module over the network to deduct stock — by deliberate design, each service writes stock changes directly to the same underlying database tables inside its own transaction, so a sale and its stock deduction always succeed or fail together. This is working as intended: the core running totals (what the valuation report and audit trail rely on) stay accurate.

The per-warehouse Stock Levels screen is the one place that can drift

A separate summary table specifically built for fast per-warehouse lookup (what the Stock Levels page reads) is only updated by this module's own background processes — not directly by a sale or a purchase receipt happening in another module. A nightly job compares the two and logs any mismatch, and a manual full-recompute is available, but there is no automatic real-time correction today. If Stock Levels ever looks persistently wrong for a specific warehouse (not just briefly after a transaction), running the manual recompute is the fix, not waiting.

Super Admin documentation

Limitations to Know

Full technical detail on all of these is in Known Issues.

  1. Inventory Manager role can't perform inventory writes — see Access & Roles. Likely your highest-priority fix if you rely on this role.
  2. New stock transfers can get stuck in Draft — no UI action to submit them.
  3. Cancelling a dispatched transfer doesn't restore stock — avoid doing this today.
  4. Cancelling a confirmed invoice doesn't restore the original cost basis (FIFO layer / WACC dilution) — quantity comes back, but the ₹ cost history doesn't reverse cleanly. A known, still-open gap from an earlier phase.
  5. Purchase returns and consignment sales don't update stock valuation — quantity is correct, but the ₹ value shown for affected items will drift over time if your business uses either of these flows.
  6. Stock adjustments have no self-service cancel once submitted.

Stock transfer lifecycle (service layer)

DRAFT → SUBMITTED → APPROVED → DISPATCHED → RECEIVED, with CANCELLED reachable from any state except RECEIVED/CANCELLED itself. dispatch() deducts source-warehouse stock and writes a STOCK_OUT ledger row immediately; receive() is a separate transaction that adds stock at the destination.

cancel() doesn't reverse a dispatch

cancel() permits cancelling a DISPATCHED transfer but never calls the ledger to restore the deducted stock — a concrete correctness bug, not a hypothetical. StockTransferService.ts:216-235

IN_TRANSIT is defined but never set

receive()'s guard accepts DISPATCHED or IN_TRANSIT, but nothing in the codebase ever transitions a transfer to IN_TRANSIT — dispatch() only ever sets DISPATCHED. Dead schema value or incomplete feature. StockTransferService.ts:112-158,167

Cross-service stock writes

By deliberate architecture decision (documented in ES-13's completion notes as an explicit "Architecture Deviation"), InvoiceService.confirm() (sales-service), GRNService.approve() and PurchaseReturnService.approve() (purchase-service), and ConsignmentService.recordSale() all write directly to the shared items and inventory_ledger tables inside their own transaction, rather than calling inventory-service's HTTP API — an HTTP call "cannot be undone if this transaction later aborts," which would break atomicity. Each of these services carries its **own copy** of a `ValuationService` for this reason (three near-identical copies exist: inventory-service, purchase-service, sales-service). apps/sales-service/src/domain/InvoiceService.ts:382-390,395-449

projection_stock_level is not touched by these direct writes

Only inventory-service's own domain code updates projection_stock_level (the table the Stock Levels page reads). Grep across apps/sales-service/src and apps/purchase-service/src for projectionStockLevel returns zero matches. A nightly reconciliation job (reconciliation.job.ts:60-116) detects and logs drift; a manual full-recompute job exists (projectionRebuildJobs.ts:54-135, cron 'manual-only') whose own comments confirm this gap is expected, not accidental.

Data Model

packages/db-client/src/schema/inventory.ts and items.ts.

items (relevant columns)

availableQty, reservedQty, costingMethod (FIFO|WACC, default WACC, set per item — not per tenant), waccCost, currentStockValue, reorderLevel/reorderQty. These counters are tenant-wide, not per-warehouse.

packages/db-client/src/schema/items.ts:158-230

inventory_ledger

Append-only. movementType: STOCK_IN|STOCK_OUT|ADJUSTMENT|TRANSFER_IN|TRANSFER_OUT|OPENING|RESERVATION|RESERVATION_RELEASE. Carries warehouseId, quantityBefore/quantityAfter, unitCost, cogsPerUnit, referenceType/referenceId.

packages/db-client/src/schema/inventory.ts:20-59

Per-warehouse valuation

FIFO: inventory_fifo_layers — one row per stock-in, warehouse-scoped since its creation, consumed oldest-received-first. WACC: inventory_warehouse_valuation (added later) — one row per (tenant, item, variant, warehouse), holding a true per-warehouse WACC alongside the tenant-wide figure on items.

FIFO was never missing per-warehouse tracking

FIFO's per-warehouse cost is computed live from the already warehouse-scoped inventory_fifo_layers table and never reads inventory_warehouse_valuation at all. The later per-warehouse valuation work was entirely about closing a WACC-specific gap — not a FIFO one, despite some older internal notes implying otherwise.

packages/db-client/src/schema/inventory.ts:62-108

Known Issues & Needs Review

Ranked by real-world impact. Everything below is a confirmed code-reading finding.

INVENTORY_MANAGER role lacks the permission nearly every write action here requires

WAREHOUSE_MANAGE gates almost all state-changing endpoints in this module (transfers, adjustments, physical verification, warehouse CRUD), but the seeded Inventory Manager role is never granted it — only Owner/Admin have it via their all-permissions wildcard. Inventory Manager does have STOCK_VIEW-adjacent and reservation permissions, none of which any route actually checks. apps/tenant-service/src/rbac/role-defaults.ts:139-164 vs. every requirePermission(PERMISSIONS.WAREHOUSE_MANAGE) call across adjustment.routes.ts, transfer.routes.ts, physical-verification.routes.ts, warehouse.routes.ts.

Cancelling a dispatched transfer doesn't reverse the stock deduction

StockTransferService.cancel() permits cancelling a DISPATCHED transfer without any ledger-reversal call, stranding deducted stock. StockTransferService.ts:216-235

projection_stock_level (Stock Levels page) isn't updated by sales/purchase writes

Only inventory-service's own domain code updates this table; sales-service and purchase-service update items/inventory_ledger directly but never this projection. A nightly reconciliation job and a manual full-rebuild job both exist specifically because of this — their own code comments confirm the gap is known, not hidden.

IN_TRANSIT transfer status is dead or incomplete

Checked as a valid pre-receive state but never set by any code path — dispatch() only ever sets DISPATCHED.

No cost-basis reversal on invoice cancellation

Self-documented, unresolved: cancelling a confirmed invoice restores quantity via a plain stock-in but doesn't restore the original FIFO layer or reverse WACC dilution.

Purchase returns and consignment sales never update valuation

Self-documented, unresolved: PurchaseReturnService.approve() and ConsignmentService.recordSale() deduct stock correctly but aren't wired into ValuationService — the ₹ value shown will drift from true remaining value over time for tenants using either flow.

Unreachable UI actions

stockTransferApi.submit()/.cancel() and stockAdjustmentApi.cancel() exist and work server-side but are never called from any component. A new transfer has no UI path out of Draft.

Unused permission constants

STOCK_VIEW, STOCK_ADJUST, STOCK_ADJUST_APPROVE, STOCK_TRANSFER, STOCK_TRANSFER_APPROVE, STOCK_PHYSICAL_VERIFY, STOCK_RESERVE, STOCK_REPORT_VIEW, WAREHOUSE_CREATE/UPDATE/DELETE, and others are defined but never checked by any inventory-service route — the coarser ITEM_VIEW/ITEM_EDIT/WAREHOUSE_VIEW/WAREHOUSE_MANAGE/REPORT_VIEW are used instead.

Test coverage gap

Ledger/valuation logic is well tested (7 test files). Zero test files reference StockTransferService, ReservationEngine, StockAdjustmentService, PhysicalVerificationService, or any of the warehouse/transfer/reservation/adjustment/physical-verification route files.

Confirmed solid: cross-service stock consistency (core tables)

Despite sales-service and purchase-service writing to items/inventory_ledger directly rather than through inventory-service's API, this is a deliberate, documented architecture choice to preserve transactional atomicity — and it works: the core running totals and audit trail stay correct. The gap is narrower than "two disconnected pictures of stock" — it's specifically the per-warehouse read-model projection, not the ledger or item counters themselves.

Confirmed solid: per-warehouse valuation for both costing methods

FIFO has been genuinely per-warehouse since its introduction; WACC was closed to the same standard by later work. Both are real, working, and distinct from the "Estimated" fallback used only when a warehouse genuinely has no movement history for an item yet.

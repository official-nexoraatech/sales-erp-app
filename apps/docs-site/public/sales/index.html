Sales & Invoicing Overview

This module covers the desktop ERP's Sales area — manual invoices, sale returns, and payments — as opposed to the POS Terminal, which is a separate app for fast counter billing. Built from a direct code audit; several workflows described in docs/training/CASHIER_GUIDE.md for this area turned out not to work as written — those are called out explicitly rather than glossed over.

Use this module for anything POS doesn't cover well: wholesale/B2B invoices, GST-heavy billing, quotations that convert into invoices, and the paperwork side of returns and receivables.

Before you rely on this module — read this first

Two workflows here are currently broken in ways that affect real business outcomes, not just convenience: Sale Returns cannot be saved from the screen at all, and every approved sale return posts to the accounting ledger as ₹0 regardless of its real value. Full detail on both is in Sale Returns & Credit Notes and Known Issues.

Who uses it

Anyone with INVOICE_VIEW/INVOICE_CREATE for invoices, PAYMENT_VIEW/PAYMENT_CREATE for payments. Sale returns are gated by INVOICE_CANCEL specifically (not a dedicated "returns" permission) — see Access & Roles for why that matters.

What's here

  • Invoices — create as a draft, confirm to finalize and deduct stock, cancel if needed. No editing after creation, at any stage.
  • Quotations & Delivery Challans — can be converted into an invoice, prefilling its line items.
  • Sale Returns — intended to restore stock and issue a credit note. Currently broken end-to-end from the UI.
  • Payments — record money received against a customer; allocating it to specific invoices is a separate step that is also currently broken from the UI.

Creating & Confirming an Invoice

Creating a draft

  1. Start a new invoice

    Sales → Invoices → New Invoice, or convert an existing Quotation or Delivery Challan (its items carry over automatically).

  2. Fill the header

    Customer, Branch, Warehouse, Invoice Date, Due Date, and Place of Supply.

  3. Add line items

    Search each item, then enter quantity, rate, discount %, and GST % yourself.
    Rate does not auto-fill

    Unlike the price-list feature used elsewhere in the app, a new line here always starts at ₹0 — you need to type the rate manually even if a price list exists for the item.

  4. Save as Draft

    This is the only save option on this screen — there's no combined "confirm and take payment" action here. A draft holds no stock and reserves nothing.

Confirming the invoice

Open the saved draft and click Confirm Invoice. You'll be asked to type the invoice number yourself.

Invoice numbers are typed, not generated

The system only checks that the number you type is unique for your tenant — it does not generate one for you and does not enforce any sequence. Two different users could type numbers out of order. If your business needs strictly sequential invoice numbers for GST audit purposes, that has to be enforced by your own process today, not the software. See Invoice Numbering & GST Compliance.

Confirming is what actually deducts stock and locks in the invoice number — a draft does neither.

Invoice dates and closed accounting periods

If the invoice date falls in a month your accountant has already closed, confirming the invoice fails with a clear error. You can still save it as a draft with that date — the check only runs at confirm time, so you won't find out until then.

You cannot edit an invoice, ever

No edit exists at any status

This is stronger than "you can't edit a confirmed invoice" — there is no edit action for a draft invoice either, once it's saved. If you need to change something, cancel it and create a new one. For a confirmed invoice, use a Sale Return instead of trying to correct the original.

Sale Returns & Credit Notes

The "New Return" form cannot currently be saved

This is a real, confirmed defect, not a training gap: the return form doesn't have a way to pick which items are being returned, and always submits an empty item list — which the server always rejects, because a return must have at least one item. Every attempt to save a new sale return fails. There is no workaround from this screen today; it needs a development fix (see Known Issues). If you need to process a return right now, do it as a manual adjustment through your accountant rather than via this screen.

How this is meant to work

Documenting the intended flow so it's ready to use the moment it's fixed:

1
Find the original invoice, select returned items and quantity
→
2
Confirm the return
→
3
Stock restored (if physical) + a Credit Note issued automatically

A returned quantity can't exceed what's left after any earlier approved returns on the same invoice line — the system tracks that cumulatively, not just against the original sale.

Applying a Credit Note

No "Apply Credit Note" button exists

Even for a credit note that already exists, there is currently no place in the app to apply it against a new invoice — despite this being a real, working capability on the server. Applying a credit note today requires a developer to call the API directly; there is no supported way to do it as a normal user.

Who can do this

Creating a sale return requires the same permission as cancelling an invoice, not a separate "create return" permission. Check with your admin if you don't see the option — see Access & Roles for why a Cashier-type account often won't.

Recording Payments

Recording a payment

  1. Sales → Payments → Receive Payment

    Search for the customer.

  2. Enter the amount and mode

    Cash, Card, UPI, Cheque, NEFT, or RTGS. For a cheque, enter the cheque number and bank name.

  3. Save

    The payment is recorded against the customer.
This does not link the payment to any invoice

Recording a payment here does not show or select which of the customer's invoices it's for — it's just logged against the customer. Linking it to specific invoice(s) is a separate step called "allocating," and that screen is currently broken: it leads to a page that doesn't exist. Right now there is no way to mark a specific invoice as paid via this flow.

Cheque payments

No "Pending Deposit" tracking

Recording a bounced cheque works and correctly reverses the payment. But there is no "Pending Deposit" or "Deposited" status anywhere in the system — every payment mode, including cheque, starts the same way as cash or card. If your process depends on tracking which cheques are still sitting undeposited, that isn't tracked here today.

"Oldest invoice first"

There's no automatic ordering logic in the system — allocating a payment (once that screen works) is a fully manual choice of which invoice(s) to apply it to, in whatever order you choose.

Access & Roles

ActionPermission actually checkedNotes

Dedicated return/credit-note permissions exist but are never checked

The permission system defines SALE_RETURN_VIEW/CREATE/APPROVE/CANCEL and CREDIT_NOTE_VIEW/CREATE/CANCEL/ADJUST, and grants some of them to Sales Manager and Cashier roles by default — but no route in the code actually checks any of them. The real gate is INVOICE_CANCEL (for returns) and PAYMENT_CREATE (for credit notes), as shown above. Practical effect: a default Cashier role does not have INVOICE_CANCEL, so a cashier cannot create a sale return today even though they're separately granted the (unused) SALE_RETURN_CREATE permission. If cashiers are expected to process returns, this needs a role-default fix, not just a training note.

Converting a quotation is Owner/Admin-only by default

QUOTATION_CONVERT isn't explicitly listed in the Sales Manager or Cashier permission sets — only Owner/Admin roles get it (via their all-permissions grant). Confirm whether Sales Manager should also be able to convert a quotation to an invoice; if so, this is a role-default gap.

Invoice Numbering & GST Compliance

Manual invoice numbers are free text, not enforced-sequential

When a user confirms an invoice from this module, they type the invoice number themselves. The system only guarantees it's unique per tenant — nothing enforces that numbers are sequential or gapless. Indian GST rules generally expect invoice numbering to be sequential and traceable. If this tenant needs to demonstrate compliant sequential numbering, that discipline currently depends entirely on staff typing numbers in order, with no system enforcement or warning if they don't.

This is a different (and weaker) guarantee than the POS terminal, which auto-generates its own invoice numbers from a timestamp — not sequential either, but at least not user-editable. Neither path currently produces a true gapless sequence; treat both as a compliance item to resolve before relying on either for statutory reporting, and see the equivalent note on the POS side.

Every sale return posts ₹0 to the general ledger

Independently of invoice numbering: a data-mismatch bug means the accounting entry generated for every approved sale return is booked at ₹0, regardless of the return's real value (see Known Issues for the technical detail). Combined with the fact that Sale Returns can't currently be saved at all, this hasn't yet caused real financial data to be posted incorrectly in practice — but it will the moment the save-form bug above is fixed, unless this is fixed at the same time.

Limitations to Know

Ranked by how much they should worry you before depending on this module for real operations. Technical detail on all of these is in Known Issues.

  1. Sale Returns cannot be saved at all — the "New Return" screen always fails. This is a launch-blocker if returns are part of your workflow, not a cosmetic gap.
  2. Sale-return accounting entries post at ₹0 — currently masked by #1 (nothing gets that far), but must be fixed before #1 is fixed, or every future return will silently understate your books.
  3. Payments can't be linked to invoices from the UI — the allocation screen is a dead link. Receivables tracking by invoice isn't currently usable end-to-end.
  4. Credit notes can't be applied from the UI — they're created correctly but have no "use this credit" button anywhere.
  5. Invoice numbering has no sequence enforcement — a GST-compliance question your accountant should weigh in on.
  6. Sales Manager may be unable to convert quotations — confirm this is intended.

Data Model

packages/db-client/src/schema/sales.ts. Invoices/invoice_lines share the same table as POS (see the POS module's Data Model page) — differences noted below.

invoices (Sales-specific columns)

quotation_id, delivery_challan_id — links unique to this flow, not present on POS-created invoices. invoice_number is nullable and unique per tenant (invoices_tenant_number), but populated from free-text user input rather than generated. client_operation_id is a POS-only concept, unused on this path.

packages/db-client/src/schema/sales.ts:92-158

payments

payment_mode: CASH|CARD|UPI|CHEQUE|NEFT|RTGS|CREDIT_NOTE|ADVANCE|LOYALTY. status: RECEIVED|PARTIALLY_ALLOCATED|FULLY_ALLOCATED|BOUNCED|REFUNDED — no PENDING_DEPOSIT/DEPOSITED values exist. Cheque columns: cheque_number, cheque_bank_name, cheque_date, bounced_at, bounce_reason.

packages/db-client/src/schema/sales.ts:266-306

payment_allocations

payment_id, invoice_id, amount, allocated_by — a plain many-to-many join. No "oldest first" or any automatic-ordering concept anywhere in the schema.

packages/db-client/src/schema/sales.ts:308-323

sale_returns / sale_return_lines / credit_notes

sale_returns.status: DRAFT|APPROVED|CANCELLED — in practice always set straight to APPROVED in the same transaction as creation, so DRAFT is a theoretical state not exercised by current code. credit_notes.status: OPEN|PARTIALLY_USED|FULLY_USED|REFUNDED, with used_amount/remaining_amount, and FKs to both the originating sale_return_id and original_invoice_id.

Known Issues & Needs Review

Ranked by real-world impact. Everything below is a confirmed code-reading finding, not a guess.

Every sale-return accounting entry posts at ₹0

SaleReturnService.create publishes the SALE_RETURN_APPROVED event with a field named totalAmount, but SaleReturnAccountingConsumer.handleSaleReturnApproved reads a field named grandTotal — which is never sent. The consumer's Number(p.grandTotal ?? 0) therefore always evaluates to 0, so every sale-return journal entry is booked at ₹0, silently understating the ledger impact of every return. No accounting-service test covers this consumer. apps/sales-service/src/domain/SaleReturnService.ts:236-244, apps/accounting-service/src/consumers/SaleReturnAccountingConsumer.ts:9-21

"New Return" can never be saved

SaleReturnsPage.tsx always submits lines: []; the backend's Zod schema requires lines.min(1). Every submission of this form fails with a 400. The UI's own hint text ("add line items by editing the return record") describes a workflow with no corresponding edit endpoint. apps/web-frontend/src/pages/sales/SaleReturnsPage.tsx:130, apps/sales-service/src/api/sale-return.routes.ts:20-25

Payment allocation is a dead route

PaymentsPage.tsx links to /sales/payments/:id/allocate; no such route is registered in App.tsx, and no matching page component exists. Falls through to NotFoundPage. The backend endpoint (POST /payments/:id/allocate) works and is tested — it's simply unreachable from the UI. apps/web-frontend/src/pages/sales/PaymentsPage.tsx:110, apps/web-frontend/src/App.tsx:297-298,391

Permission model mismatch

Dedicated SALE_RETURN_*/CREDIT_NOTE_* permissions are defined and granted by role defaults but never checked by any route — actual gating reuses INVOICE_CANCEL/PAYMENT_CREATE. Net effect: default Cashier lacks INVOICE_CANCEL and so cannot create a sale return, despite separately holding the (unused) SALE_RETURN_CREATE permission. packages/shared-types/src/permissions.ts:109-118, apps/tenant-service/src/rbac/role-defaults.ts:37-41,59-74

No sequential invoice numbering on the manual path

The confirm step accepts any unique free-text string as the invoice number — no auto-generation, no gap/sequence check. A GST-compliance question, not just a UX one. apps/sales-service/src/api/invoice.routes.ts:54-56, InvoiceService.ts:346-356

QUOTATION_CONVERT not granted to Sales Manager

Only Owner/Admin (via their all-permissions wildcard) can convert a quotation to an invoice by default. May be intentional; confirm with product. apps/sales-service/src/api/quotation.routes.ts:130-131, role-defaults.ts:14-23,25-74

Unconsumed event

CREDIT_NOTE_CREATED is published on every sale return but is never subscribed to by any consumer — its only accounting effect flows through the (buggy, ₹0) SALE_RETURN_APPROVED handler instead. apps/sales-service/src/domain/SaleReturnService.ts:246-254

Dead client API surface

saleReturnApi.applyCreditNote/refundCreditNote exist in the frontend API client but are called from nowhere in any component. apps/web-frontend/src/api/endpoints.ts:503-504

Test coverage gaps

Good coverage exists for period-closure, credit-limit/price-floor, cancel+stock-restore, and return-quantity-exceeded checks. Zero test coverage found for SaleReturnService.applyCreditNote, PaymentService.bounceCheque, or the credit-note refund endpoint — consistent with these being the least-exercised parts of the flow. No TODO/FIXME comments exist in any reviewed file; none of the gaps above are flagged in-code.

Confirmed solid, once reachable

The underlying service logic for sale returns and credit notes is functionally correct at the code level — cumulative return-quantity validation, conditional stock restoration for physical returns, automatic credit-note creation, and race-safe credit-note application (row-locked against concurrent payment allocation) all check out. The defects above are UI-reachability and event-payload bugs sitting on top of a reasonable domain layer, not a fundamentally broken design.

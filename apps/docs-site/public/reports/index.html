Client documentation

Reports Overview

Ninth module documented from a direct code audit. Covers the general-purpose Reports browser — a single screen that can run any of 77 registered reports across Sales, Purchase, Inventory, Financial, HR, GST, and Analytics.

This is separate from the dedicated Trial Balance / Profit & Loss / Balance Sheet / Cash Flow pages documented under Accounting & GST — those four also happen to appear in this general browser too, computed by a second, independently-built copy of the same logic. They agree with each other except for Cash Flow, where the version in this browser is missing the Investing/Financing breakdown the dedicated page has — see that module's "Two Cash Flow Reports That Disagree" page for detail.

Good news correcting an older internal note

An older audit flagged the "Day Book" report (and several others) as having broken column references. That's been fixed — Day Book, Account Ledger, Expense Analysis, Bank Book, and Fund Flow all now query the correct real columns and are covered by passing tests. If you're consulting older notes on this area, treat that specific concern as resolved.

Exporting to CSV or Excel silently produces a broken file for many of the busiest reports

For reports that process in the background (most of the higher-volume ones — sales register, purchase register, stock movement, payroll, GST register, and more), clicking Export CSV/Excel downloads a tiny file containing a status message instead of your actual data — with no error shown. See Exporting Reports before relying on an exported file from one of these.

Browsing & Running a Report

  1. Find a report

    Search by name or filter by category on the Reports page.

  2. Set parameters

    Date range (defaulting to this month), and any report-specific filters like branch or warehouse.

  3. Run it

    Results display as a table, with totals at the bottom for currency/number columns. Busier reports process in the background and update automatically once ready — a quick "waiting" state is normal for those.
The report list shows you reports you might not actually be allowed to run

Every report you have general reporting access to appears in the list, regardless of whether you hold that specific report's own permission — you'll only find out you're blocked after clicking in and trying to run it. Seeing a report in the list isn't a guarantee you can use it.

Exporting Reports

Two export formats exist: CSV and Excel. There is no PDF export from this screen.

CSV/Excel export doesn't work for background-processed reports

About 30 of the 77 reports — generally the higher-volume ones, including Sales Register, Sales by Customer/Item, Customer Ledger, Purchase Register, Purchase by Item, Supplier Ledger, GRN Report, Stock Summary/Movement/Ageing, Inventory Valuation, Account Ledger, Bank Book, Payroll Report, Attendance Report, and GST Register — process in the background. For exactly these reports, clicking Export CSV or Export Excel currently downloads a small file containing a status message instead of your report data, with no error or warning shown. Always open the report on screen and verify it looks right before trusting an exported file from one of these — export works correctly for the remaining ~54 reports that don't process in the background.

Scheduling Reports

A scheduled report emails itself to chosen recipients on a recurring cadence, with a CSV or Excel attachment of the data.

Do not choose "PDF" as the schedule format

PDF is offered as a format option, but it isn't actually implemented — a PDF-format schedule sends an email with no attachment at all, while the email text still (incorrectly) says "see the attached file for the full report data." Always choose CSV or Excel for a scheduled report; both of those genuinely attach real data.

Scheduling is currently Owner/Admin-only

No other role can create or manage a scheduled report by default — see Access & Roles.

Super Admin documentation

Access & Roles

Several roles can't run the reports most relevant to their own job

Each report declares its own required permission, and several of those permissions aren't granted to the role you'd expect to use them — reachable only by Owner/Admin-tier accounts today:

  • HR Manager cannot run the Attendance Report (a different permission than the attendance-marking rights it does have).
  • Inventory Manager cannot run the Stock Transfer, Stock Adjustment, Physical Verification, Fabric Roll, or Reorder reports — despite having the underlying permissions to perform those actions themselves.
  • Accountant cannot run the Bank Book, GSTR-1, or GSTR-3B reports.
  • No one except Owner/Admin can create a scheduled report, or open the Dashboard/Analytics screens at all.

If staff report being unable to run a report you'd expect their role to use daily, this is very likely why.

Report Data Freshness

Reports (and dashboards) are architected to read from a separate database copy rather than the live transactional database — this is the only part of the system built this way, done deliberately so heavy reporting queries can't slow down checkout, invoicing, or any other live operation.

In practice today, this doesn't cause any noticeable staleness

The safeguard that's supposed to detect when that separate copy is lagging behind falls back to the live database automatically if it can't confirm the copy is healthy and current — and in this environment's own local/development setup, that copy isn't genuinely kept in sync in the first place, so the fallback triggers every time and reports quietly read live data regardless. In a fully production-configured deployment with real replication actually set up, reports could lag by roughly ten seconds and dashboards by up to two minutes behind live activity — an intentional tradeoff for performance, not a bug — but that real setup has not yet been confirmed as done per the project's own deployment notes. Confirm this explicitly before assuming reports reflect live data with any specific staleness guarantee in production.

Limitations to Know

Full technical detail is on Known Issues.

  1. CSV/Excel export is broken for ~30 background-processed reports — the most commonly needed ones for external analysis.
  2. PDF-format scheduled reports silently send no attachment.
  3. Several operational roles can't run reports core to their own job — a recurring role-configuration gap, same pattern found in every other module.
  4. Real database replication for reports has not been confirmed as actually provisioned in production — the routing code exists and is tested, but the deployment step itself is marked incomplete in the project's own notes.
  5. The report list doesn't pre-filter by what you can actually run — expect some access-denied surprises.

Developer documentation

Architecture

Frontend: apps/web-frontend/src/pages/reports/ReportsPage.tsx (browse/search/filter) → ReportViewerPage.tsx (run one). Backend: apps/report-service/src/domain/ReportRegistry.ts (77 ReportDefinition entries: slug/name/category/description/params/columns/permission/supportsAsync) + ReportEngine.ts (one SQL-generating case per slug, ~2000 lines).

Registry ↔ engine parity is clean

Every registry slug has a matching engine case and vice versa — confirmed by an exhaustive diff. This module doesn't have the "dead API surface" pattern found elsewhere.

Sync vs. async execution

23 of 77 reports are flagged supportsAsync: true (the higher-volume ones) and are always queued as a background job server-side regardless of what the client requests — returning 202 {runId, status:'PENDING'} immediately. The frontend polls a run-history endpoint every 1.5s until the job completes. The remaining 54 reports run synchronously and return results inline.

apps/report-service/src/api/analytics-reports.routes.ts:84; apps/web-frontend/src/pages/reports/ReportViewerPage.tsx:115-142

The CSV/Excel export bug, precisely

The frontend's export buttons POST directly to /api/v2/reports/:slug/run with a format field and always treat the response as a downloadable file blob. On the backend, the supportsAsync branch is checked before the format branch — so for any of the 23 async reports, the response is always the small {runId, status, message} JSON object, which gets saved to disk with a .csv/.xlsx extension and JSON content inside. Export only works end-to-end for the 54 synchronous reports. No test exercises this path — all existing tests call ReportEngine.generate() directly, bypassing the route handler where this bug actually lives. analytics-reports.routes.ts:84,139-153, ReportViewerPage.tsx:144-163

Dashboards are a separate feature

The three "Analytics" links on the Reports page (Sales/Inventory/HR) go to dedicated chart dashboards backed by dashboard.routes.ts and daily KPI projection tables — a distinct code path from the registry-driven report runner, gated on its own DASHBOARD_VIEW permission (also Owner/Admin-only by default).

Data Model

Reports mostly query real transactional/CQRS tables already documented under their owning modules (invoices, projection_stock_level, inventory_ledger, payroll_slips, financial_entries, etc.) rather than maintaining their own copies of data. This module's own schema is limited to scheduling/run-tracking infrastructure.

Report schedules & run history

packages/db-client/src/schema/report.ts defines the report-schedule table (report slug, recipients, cadence, format — defaulting to PDF, which is the format that doesn't actually work, see Known Issues) and a run-history table tracking each execution's status (PENDING/COMPLETED/FAILED) for the async-report polling flow.

Read-replica routing (report-service only)

A ReplicaRouter health-checks a Postgres read replica (via pg_last_xact_replay_timestamp() lag) and falls back to the primary database automatically if the replica is unhealthy or its lag exceeds a threshold — 5 seconds for reports, 120 seconds for dashboard KPIs. This exists only in report-service; every other service in the system reads and writes the primary database directly.

packages/db-client/src/replica-router.ts:17-46; packages/db-client/src/replica-health.ts:11-22; apps/report-service/src/main.ts:33-36

Known Issues & Needs Review

Ranked by real-world impact. Everything below is a confirmed code-reading finding unless marked Needs Review.

CSV/Excel export broken for all 23 supportsAsync:true reports

The route's async-vs-format branch order means the response for these reports is always the small job-status JSON object, saved with a .csv/.xlsx extension. Affects sales-register, sales-by-customer, sales-by-item, customer-ledger, payment-collection-report, discount-report, purchase-register, purchase-by-item, supplier-ledger, grn-report, supplier-payment-report, stock-summary, stock-movement, inventory-valuation, stock-ageing, warehouse-wise-stock, stock-ledger, account-ledger, bank-book, journal-report, payroll-report, attendance-report, and gst-register. analytics-reports.routes.ts:84,139-153, ReportViewerPage.tsx:144-163

Scheduled reports in PDF format send no attachment

ReportFormatter only implements toCSV()/toExcel() — no toPDF() exists. ScheduledReportJob.runSchedule()'s format branch handles only EXCEL/CSV; for PDF neither fires, so the email sends with no attachment at all — while its body text still claims "see the attached file." The schema's own default format is PDF, and the frontend's schedule-creation UI offers it as a selectable option, making this an easy trap. Not covered by any test (existing scheduled-report tests only exercise the CSV path). ReportFormatter.ts, ScheduledReportJob.ts:146-158,173,179

Permission gaps block several roles from reports core to their job

ATTENDANCE_REPORT (HR Manager), STOCK_TRANSFER_VIEW/STOCK_ADJUSTMENT_VIEW/PHYSICAL_VERIFICATION_VIEW/FABRIC_ROLL_VIEW/REORDER_VIEW (Inventory Manager, which has the underlying action permissions but not the report-view ones), GSTR1_VIEW/GSTR3B_VIEW/BANK_RECONCILIATION_VIEW (Accountant), TDS_VIEW, CRM_LOYALTY_VIEW, ALTERATION_VIEW, REPORT_SCHEDULE, and DASHBOARD_VIEW are all granted to no named business role — Owner/Admin/Super Admin only.

Production read-replica setup for reports is unconfirmed

The routing/fallback code is real and unit-tested, but the project's own deployment notes mark provisioning a genuine streaming read-replica (and a read-only DB role for it) as an incomplete checklist item. In this repo's local dev configuration, the "replica" isn't true streaming replication, so the health check always fails and every report silently falls back to reading the primary database — meaning the staleness tradeoff has never actually been exercised against a real replica in this environment.

Report list isn't pre-filtered by the viewer's specific permissions

All 77 reports are shown to anyone with the blanket REPORT_VIEW permission; a report-specific 403 only appears after attempting to run one. A minor UX gap, not a security issue — the backend correctly blocks the run itself.

GSTR-1 report's GST-rate column is hardcoded to 0

Every other column is computed from real data; this one specific column is a placeholder.

Scheduled-report email delivery bypasses notification-service

Unlike the rest of the application, which routes outbound email/SMS/WhatsApp through notification-service, scheduled reports send email directly via a local mail library. Not currently broken, but an inconsistent integration pattern worth normalizing.

Dead declarations

REPORT_EXPORT permission is granted to five roles but never checked anywhere. A multiselect parameter type is declared in the report-definition interface but used by no report and unhandled by the viewer UI. A format state variable in the report viewer is set but never read — the "Run Report" button always requests JSON regardless.

Confirmed fixed since an earlier internal note: the "broken columns" reports

Day Book, Account Ledger, Expense Analysis, Bank Book, and Fund Flow were previously flagged as referencing non-existent database columns. All five now use the correct real column names and are covered by passing regression tests (verified by actually running the test suite: 122/122 passing). If you're consulting older notes, treat this specific class of bug as resolved for these five reports.

Confirmed solid: registry/engine parity and tenant isolation

Every one of the 77 registered reports has exactly one matching implementation, with no orphans in either direction — unlike the dead-API-surface pattern found in several other modules. Tenant isolation is exhaustively tested across all 77 report types, confirming no report can ever leak another tenant's data through its generated SQL.

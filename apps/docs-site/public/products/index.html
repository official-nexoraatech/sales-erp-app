Products & Variants Overview

Fifth module documented from a direct code audit. Covers items, categories, brands, units, barcode labels, and price lists.
Despite the module's name, there is currently no way to create a "variant" (size/color) at all

The system has real database support for per-item variants and even a flexible size/color attribute system underneath — but neither has a screen anywhere in the app. Every item today is managed as a single, non-variant record. If your business genuinely needs to track "Red, size M" separately from "Blue, size L" as distinct stock lines, that isn't achievable through the UI today — see Variant & Size/Color Tracking for what this means in practice and a same-item workaround.

What's here

  • Items — the product master: name, code, GST/HSN, pricing, barcode.
  • Categories, Brands, Units — flat lists used to classify items.
  • Barcode Labels — generate and print real barcode/QR labels for items.
  • Price Lists — create a named price list; see below for a significant caveat.
Price Lists don't currently affect any price you charge

You can create a price list, but there's no screen to put prices into it, and neither POS nor the Sales invoice screen reads from a price list even when one exists. Don't rely on this feature for differentiated pricing yet — see Price Lists.

Creating & Managing Items

  1. Basic Information

    Name, Item Code (SKU), Category, Brand, Unit of Measure, Description.
    "Auto-generated if blank" doesn't happen

    The Item Code field hints that it will be generated for you if left empty — it won't be. Leaving it blank saves the item with no code. Type your own code if you rely on item codes anywhere downstream (labels, reports, integrations).

  2. GST & HSN

    Search and pick an HSN code, set the GST rate (0/5/12/18/28%) and Cess rate.

  3. Pricing

    MRP, Sale Price, Minimum Sale Price (a hard floor — see Sales & Invoicing's pricing rules), Purchase Price.

  4. Inventory & Barcode

    Track Inventory toggle, Fabric Item toggle (reveals a Fabric Width field for items sold by length), Reorder Level, and a manual Barcode field with a type (EAN13/CODE128/QR/Custom).

Editing an item works the same form; every save is recorded in a price/change history you can view per item, including plain edits and specifically-flagged price changes.

Deleting an item is a soft delete (it's marked Discontinued, not erased) and is blocked if the item already has stock movement history.

Categories, Brands & Units

All three are simple named lists you attach to items — Name, Code, and a short Description.

No sub-categories

The system can technically store a category inside another category, but there's no way to set that from the Categories screen — every category is created as a flat, top-level entry today.

Deleting a category or brand doesn't check whether items still use it

Nothing stops you from deleting a category or brand that items still reference. Nothing breaks — those items simply show a blank Category/Brand afterward — but there's no warning at delete time. Double-check nothing is still using a category or brand before removing it.

Units also cannot be deleted once created (no delete option exists) — only added or edited.

Barcode Labels & Printing

There are two, separate barcode features in the app — easy to confuse, so here's the distinction:

Barcode Labels (Production → Barcode Labels)

This is the one to use for printing physical labels. Search for an item, choose a quantity and symbology (EAN13, CODE128, or QR), and generate a batch — each one renders as a real, scannable barcode/QR image on screen, ready to print via your browser's print dialog.

This actually works

Barcode generation and printing here is fully functional today, including a correct EAN13 check digit — this is a genuinely shipped feature, not a stub.

The "Generate Barcode" button on the Items list

A different, simpler action: it writes a single barcode value directly onto that item's own record and shows it in a confirmation message. It does not open a printable label or produce an image — for an actual printable label, use Barcode Labels above instead.

These two features don't share data

Generating a barcode one way doesn't create anything the other way recognizes. If you use both features on the same item, you may end up with two different barcode values in two different places.

Price Lists

Price lists are currently empty containers with no effect on pricing

You can create a price list (name, currency, valid dates) — but there's no screen to add specific item prices into it, and even if prices existed inside one, neither POS nor the Sales invoice screen would read from it. A sale always uses the item's own Sale Price (POS) or whatever the cashier/clerk types manually (Sales invoices, which don't even default to the item's Sale Price — see Sales & Invoicing). If you were expecting customer-tier or bulk pricing from this screen, it isn't functional yet.

Until this is built out, manage differentiated pricing outside the system (a spreadsheet or a manual process) rather than relying on a price list here.

Variant & Size/Color Tracking

Stock is not tracked per size/color — only per item

This is the single most important thing to understand about this module for a clothing business: even where the database has room to record variants, every stock number the system actually enforces — available quantity, reservations, oversell prevention — is pooled at the item level, not broken out by size or color. In practice, that means if "T-Shirt — Red" and "T-Shirt — Blue" were ever tracked as the same item, selling the last unit of Red would be allowed as long as Blue units are still in stock, because the system only sees one combined number.

What this means for how you set up products today

Since there's no working variant screen (see Overview), the practical way to track different sizes/colors as separate, sellable stock is to create them as separate items — e.g. "T-Shirt Red M" and "T-Shirt Blue L" as two distinct item records, each with its own barcode/SKU and its own stock count. This is more manual than a true variant system, but it's the only setup that gives you accurate, size/color-specific stock levels today.

Costing method (FIFO/WACC) can't be changed

Every item is costed using the weighted-average method by default, and there is currently no field anywhere — on creation or edit — to switch an item to FIFO costing, despite FIFO being a fully supported method elsewhere in Inventory's valuation reporting. If your business needs FIFO costing for specific items, that requires a direct data change today, not a settings toggle.

Limitations to Know

Full technical detail is on Known Issues.

  1. No variant (size/color) UI exists at all — the module's namesake feature isn't usable; work around it with separate items per variant.
  2. Stock is pooled at the item level, not per variant — a real oversell risk if you ever do get variants set up via a workaround or future update.
  3. Price Lists have no effect on any real sale — don't build a pricing strategy around this screen yet.
  4. Inventory Manager can't access Price Lists — a role-configuration gap, same class already fixed once for barcodes.
  5. Item Code "auto-generate" doesn't work — type your own codes.
  6. Category/brand deletion doesn't warn about items still using them.

Architecture

Frontend: apps/web-frontend/src/pages/items/ (Categories/Brands/Units/Items/Price Lists) and src/pages/production/BarcodeLabelsPage.tsx. Backend: apps/inventory-service/src/api/{item,category,brand,unit}.routes.ts plus apps/production-service/src/api/barcode.routes.ts.

inventory/items, /new, /:id/editItemsPage, ItemFormPageITEM_VIEW/CREATE/EDIT
inventory/price-listsPriceListsPagePRICE_LIST_VIEW
production/barcode-labelsBarcodeLabelsPageBARCODE_GENERATE
No variant or attribute-set page anywhere

Exhaustive grep across apps/web-frontend/src for "variant"/"attribute" turns up only the unrelated variant="..." UI-styling prop used on buttons/badges — confirming no such screen exists, not just that it's hard to find.

Two independent barcode systems

(1) inventory-service: a single items.barcode/itemVariants.barcode value, settable manually in the item form or via the Items list's "Generate Barcode" action (deterministic EAN13 checksum, item.routes.ts:429-443). (2) production-service: batch-based barcodes/barcode_batches tables (many codes per item, deactivatable), used exclusively by BarcodeLabelsPage.tsx, which renders real SVG barcodes/QR codes client-side via jsbarcode/qrcode and opens the browser print dialog. Generating via one system doesn't touch the other's tables.

Barcode lookup fallback order differs between the two systems

production-service's GET /items/by-barcode/:value (used by POS scanning) checks its own barcodes table first, then falls back to items.barcode — it never checks item_variants.barcode. inventory-service's separate (frontend-unused) GET /items/by-barcode/:barcode checks item_variants first, then items. Needs Review: unclear if this divergence is intentional. apps/production-service/src/domain/BarcodeService.ts:168-197, apps/inventory-service/src/api/item.routes.ts:93-114

The barcode-rendering gap noted in an earlier internal audit has since been closed

An older note flagged barcode value-generation as real but rendering as entirely missing (JSON-only print endpoint, a stubbed inventory-service route referencing an uninstalled library). As of the current codebase, BarcodeLabelsPage.tsx genuinely renders and prints real barcode/QR images client-side — confirmed via package.json dependencies and the component's own render logic, and via git history showing this was added/fixed in a recent commit. Worth knowing if you're cross-referencing older internal notes on this area.

Data Model

packages/db-client/src/schema/items.ts.

items

itemCode, barcode, barcodeType (EAN13|CODE128|QR|CUSTOM), hasVariants + variantAttributeIds (jsonb array), availableQty/reservedQty (item-level, tenant-wide counters), costingMethod (default WACC, no code path ever writes a different value), waccCost, currentStockValue. Unique on (tenantId, itemCode) and (tenantId, barcode) — Postgres allows multiple NULLs through a unique index, so several items can share a blank item code simultaneously.

packages/db-client/src/schema/items.ts:158-230

itemVariants

sku (required, manual, no generator), barcode, attributeCombination (freeform jsonb Record<string,string> — not backed by the structured attribute tables below), mrp/salePrice/purchasePrice, imageUrl, isActive. No availableQty/reservedQty columns at all — variants have no first-class live stock counter; every stock mutation path in InventoryLedgerService keys its atomic update by itemId only, never variantId. A separate read-model table (projectionStockLevel) does track a per-variant number, but it's a reporting artifact the deduct/reserve path never consults — so oversell protection is effectively pooled across all of an item's variants.

packages/db-client/src/schema/items.ts:233-262

attributeSets / attributes / attributeValues — dead schema

A full flexible-attribute system exists (e.g. attribute set "Clothing" → attribute "Size"/"Color" → values "S"/"M"/"Red" with a hex color) — but no CRUD route exists for any of these three tables anywhere in inventory-service, and no frontend page references them. The only live connection is the attributeSetId foreign key on items, which nothing currently sets meaningfully.

packages/db-client/src/schema/items.ts:89-155

categories / brands / units

categories.parentId is self-referencing and indexed — real hierarchy support at the schema level, unused by any UI. brands and units are flat. Category/brand deletion is soft (sets deletedAt) with no check for referencing items; items.categoryId/items.brandId have no FK constraint, so a deleted category/brand simply renders as blank on existing items going forward.

packages/db-client/src/schema/items.ts:16-86

priceLists / priceListItems

priceListItems can price an itemId + optional variantId combination distinctly, at the schema level — never populated in practice since no UI writes to it and no consumer (POS, Sales invoicing) reads from it.

packages/db-client/src/schema/items.ts:283-329

Known Issues & Needs Review

Ranked by real-world impact. Everything below is a confirmed code-reading finding.

Variant-level stock tracking doesn't exist despite the module's name

itemVariants has no availableQty/reservedQty columns; every atomic stock mutation in InventoryLedgerService (addStock, deductStock, adjustStock, transferStock) keys its guarded update by itemId only. A per-variant number exists only in a reporting projection the deduct path never checks. Confirmed, high business impact for a clothing ERP selling by size/color.

Price lists have no populate UI and no consumer

PUT /price-lists/:id/items works server-side but has zero frontend callers — PriceListsPage.tsx can only create empty list headers. Separately, neither InvoiceService (Sales) nor POS ever queries priceLists/priceListItems at all — Sales invoice lines default to ₹0 regardless, and POS uses the item's own salePrice directly. Price lists are fully-modeled but entirely inert.

No variant creation/attribute UI anywhere

POST /items/:id/variants and the entire attributeSets/attributes/attributeValues schema have no frontend surface at all — confirmed by exhaustive grep, not just difficulty finding it.

Inventory Manager lacks PRICE_LIST_VIEW

Blocks the role that owns this module from its own Price Lists screen — the same bug class already fixed once for BARCODE_* permissions via migration 0026_pos_barcode_permission_backfill.sql, whose own comment documents the root cause precisely. This price-list instance remains unfixed.

Item Code "auto-generated if blank" is not implemented

ItemFormPage.tsx's hint text isn't backed by any generator in POST /items — whatever is submitted (including nothing) is persisted as-is.

costingMethod is never settable through any code path

Not present in the item create/update Zod schemas or the form — every item is permanently WACC-costed at the DB default, extending the same finding from the Inventory module's audit.

Category hierarchy is schema-only

categories.parentId is real and indexed, but no create/edit form ever sends it — categories are managed as a flat list in practice.

Two barcode-lookup implementations disagree on fallback order

production-service's lookup (used by POS) checks its own batch table then items.barcode, never item_variants.barcode; inventory-service's separate, frontend-unused lookup checks variants first. Unclear if intentional.

Dead permission constants

ITEM_UPDATE, ITEM_PRICE_UPDATE, ITEM_IMPORT, ITEM_EXPORT, ITEM_BARCODE_PRINT, PRICE_LIST_CREATE/UPDATE/DELETE — confirmed dead by the repo's own allowlisted regression test.

Test coverage gap

Item CRUD, optimistic locking, and concurrent item-level stock deduction are well tested. No tests cover variant creation, categories, brands, units, or price-list routes; barcode tests only cover the pure checksum function, not routes/permissions/print-data assembly.

Confirmed fixed since an earlier internal note: barcode label rendering

A prior audit (6+ days before this one) found barcode value-generation real but rendering entirely missing. Current code shows this was genuinely fixed in a recent commit — BarcodeLabelsPage.tsx now renders and prints real barcode/QR images client-side via real jsbarcode/qrcode dependencies. If you're consulting older internal audit notes on this area, treat this specific finding as resolved.

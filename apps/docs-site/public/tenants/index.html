Overview — Two Different "Super Admins"

Tenth module documented from a direct code audit. Unlike every module so far, this one isn't a workflow any tenant's staff uses — it's the layer that manages the tenants (customer businesses) themselves.
Read this before anything else on this page — two unrelated things share a confusing name

The word "Super Admin" refers to two completely different things in this system, and mixing them up will misdescribe how access control actually works:

  • Platform Operator — the real, cross-tenant SaaS-operator role. This is you, running the platform for all customer businesses. It can suspend/close any tenant, but that's essentially all it can do across tenants (see below).
  • "Super Admin" role — just the name of a top-tier staff role seeded into every individual tenant, with the same permissions as that tenant's Owner/Admin. It has zero power outside that one tenant. A clothing store's own "Super Admin" account is not the same thing as you, the platform operator.

What the Platform Operator role can actually do today, precisely: create, suspend, reactivate, or permanently close a tenant, and view basic usage figures. That's the entire cross-tenant surface that exists in the product right now — there's no cross-tenant user management and no cross-tenant audit-log viewer (see below).

End-to-End Client Journey

A map of the whole loop — onboarding a business, that business running its day-to-day operations, and what it sees when it looks back at its own numbers. This page doesn't introduce new findings; every stage links to the module that documents it in full, including whatever gaps or bugs are known there. Think of this as the table of contents for the entire site, arranged as a story instead of a list.

Every one of these actions — a sale, a goods receipt, a payroll run — quietly does a second job in the background: it writes the accounting entry that will show up in Stage 4, at the same moment it happens, not as a separate manual step.

This is a loop, not a straight line

Stage 5 isn't the end — the decisions it produces become tomorrow's Stage 3 actions. A new business only ever goes through Stage 1 once; everything from Stage 2 onward repeats for as long as the business operates.

Managing Tenant Lifecycle

Provisioning
New tenant being set up
→
Active
⇄ Suspended (reversible, either direction)
→
Closed
Permanent — no way back
Creating a tenant now has a screen

The tenant list has a "+ New Tenant" button (visible if you hold the platform-management permission) that opens a form for the organization's details, its plan, and its first Owner account — no developer/API call needed for this step anymore.

The new admin's password still isn't delivered by the system itself

The form requires you to type the new tenant's Owner password directly — there's still no auto-generated password, no "reset on first login" flag, and no email carrying the credential. You still need to relay that password to the client yourself, outside the system, after creating them.

How the new user actually logs in

This app's login screen asks for three things, not the usual two: email, password, and a numeric Tenant ID. The same email address can exist in more than one business at once, so the Tenant ID is what tells the system which business you mean.

The Tenant ID is only ever shown once, right after creation

When you submit the New Tenant form, the very next screen shows the new Tenant ID and the admin's login email — that's the only moment this ID is surfaced anywhere. If you close that screen without writing it down, you'd previously have had no way to find it again. The tenant list now also shows a Tenant ID column for exactly this reason — use it to look up an existing business's ID later if you ever need to.

What to give the client, all three together: their Tenant ID (from the success screen or the tenant list), the email you set as their contact/admin email, and the password you typed into the form. All three are required — any one missing and they can't sign in.

Suspending or closing a tenant

Both require a reason (at least 5 characters); closing additionally requires typing a literal confirmation phrase to guard against an accidental click. A closed tenant cannot be reopened — treat Close as genuinely final, unlike Suspend which is fully reversible.

Suspension isn't instant everywhere at once

The service handling the request you're using enforces the suspension immediately. Every other backend process independently re-checks within about a minute — so for a short window (under a minute), a suspended tenant's staff might still be able to complete an action already in progress elsewhere in the system. This is a real, if narrow, timing gap.

A tenant's own staff experience suspension the next time any of their screens makes a request — they're redirected to an "Account Suspended" page, not instantly logged out of a page already open.

Feature Flags

Feature flags turn specific capabilities on or off — genuinely checked live by the system, not just cosmetic. There's a platform-wide default for each flag, and any individual tenant can be given its own override.

This screen manages a tenant's own flags, not a cross-tenant control panel

The Feature Flags screen a tenant's staff use only ever shows and edits that tenant's own settings — there is currently no separate platform-operator screen for viewing or changing flags across multiple tenants at once.

One flag on this screen does nothing

"FIFO Valuation" appears as a toggle here, but nothing in the system actually checks it — whether an item uses FIFO or weighted-average costing is decided per item elsewhere (see Products & Variants), not by this flag. Toggling it currently has no effect.

Several real flags show "No description available"

The flags that actual business logic checks day-to-day (POS enabled, loyalty program, tailoring/alterations, and others) aren't in the screen's description list, so they show a generic placeholder instead of a helpful label. They still work correctly — the missing description is a documentation gap in the UI itself, not a functional one.

Impersonating a User

Impersonation lets an authorized account temporarily see the system exactly as another user would — useful for troubleshooting a specific staff member's issue.

  • Only works within the same business — you can only impersonate someone in your own tenant, never a customer of a different business, and (as the platform operator) you cannot impersonate a customer tenant's user at all today.
  • Automatically ends after one hour.
  • Fully logged — both the start and the end of every impersonation session are recorded, including who did it, who was impersonated, and why.
  • A visible banner with a live countdown and a "Stop impersonating" button stays on screen the entire time.
Stacking impersonation sessions isn't blocked on the server

The app's own interface won't let you start a second impersonation session while already impersonating someone — but that protection lives only in the browser, not the server. Treat this as a real, if narrow, gap rather than an impossible scenario.

Plans & Billing

Three plans exist — Starter, Growth, Enterprise — each with a user limit, a branch limit, and a set of feature flags that get switched on automatically for a tenant on that plan.

User and branch limits are shown but not enforced

A tenant's plan limits are genuinely recorded and displayed, but nothing currently stops a Starter-plan tenant from adding more users or branches than their plan allows. If limiting usage by plan matters to your business model today, it needs to be enforced manually, not automatically.

Feature-flag entitlements do work

Unlike the user/branch limits, the set of features a plan unlocks genuinely gets applied to a tenant when they're provisioned on that plan — this part of "plan" is real.

No way to change a tenant's plan after it's created

A plan is set once, at provisioning time, via the same developer-only process used to create the tenant in the first place. There's no upgrade/downgrade action anywhere in the app today.

There is no live payment collection, invoicing automation, or dunning process yet — the groundwork (a billing-invoice table, a "next billing date" field) exists, but nothing acts on it automatically today.

Access & Roles

Tenant management itself is correctly locked down — a rare clean bill of health

Unlike the recurring role/permission mismatches found in nearly every other module of this ERP, tenant creation/suspension/activation/closing is gated correctly and consistently, and no ordinary tenant-level role can reach these endpoints. This wasn't always true — a prior version of the system had effectively no protection here at all, and it's since been fixed and locked in with a dedicated regression test suite.

Only the Platform Operator role (or a tenant's own Owner-equivalent role, but scoped strictly to that one tenant) has any of this power. Impersonation follows the same tenant-scoping rule: it requires a specific permission, and only within one's own tenant.

Provisioning Reliability

A failed tenant setup can leave a permanently broken, half-created tenant

Setting up a new tenant involves about ten separate steps — creating the tenant record, seeding its roles, creating its first admin user and head-office branch, setting up its file storage, its search indexes, its feature flags, and more. These steps are not wrapped together as a single all-or-nothing operation. If any one step fails partway through (for example, a storage-provisioning outage), everything before that point is already permanently saved — the tenant is left stuck in a "setting up" state with no automatic retry or rollback. Cleaning this up currently requires a developer to intervene directly in the database. If you're provisioning tenants at any real volume, budget for this as a known operational risk, not an edge case that won't happen.

Limitations to Know

Full technical detail is on Known Issues.

  1. No UI to change a tenant's plan after creation — creating a tenant now has a screen, but upgrading/downgrading an existing one still requires direct backend access.
  2. Provisioning can fail partway and leave permanent orphaned state — no resume or rollback mechanism exists.
  3. Plan user/branch limits aren't enforced — display-only figures today.
  4. No cross-tenant user management or audit-log view — the platform operator's cross-tenant power is limited to tenant lifecycle and basic usage figures only.
  5. Tenant-suspension enforcement has a sub-minute cross-process staleness window.
  6. No billing/payment automation exists yet — the schema is ready; nothing acts on it.

Developer documentation

Architecture

No separate Super Admin portal exists — everything lives in apps/web-frontend, permission-gated like any other screen, under a "Platform Admin" nav group.

PLATFORM_OPERATOR vs. SUPER_ADMIN — the precise distinction

PLATFORM_OPERATOR is a real cross-tenant role, scoped to a reserved platform-operations tenant seeded by migration, holding exactly one permission: PLATFORM_TENANT_MANAGE. SUPER_ADMIN is a per-tenant system role name seeded into every tenant with the same permission set as OWNER — PLATFORM_TENANT_MANAGE is explicitly excluded from every tenant-scoped role's permission set, including this one. Do not conflate the two when reasoning about access control.

Impersonation has no dedicated page — it's a row action inside the ordinary Users screen (UsersPage.tsx), gated on IMPERSONATE_USER, with a confirmation dialog and a persistent status banner as separate shared components.

Tenant-status cache: no cross-service invalidation broadcast

An inconsistent caching pattern within the same codebase

Every service checks tenant status via a 60-second, per-process in-memory cache. The service handling the actual suspend/close request invalidates its own cache immediately — but nothing broadcasts that invalidation to any other running service instance. This is a real, if narrow, gap: contrast with the feature-flag cache below, which correctly solves the identical problem via Redis pub/sub. Worth aligning both to the same pattern. packages/platform-sdk/src/tenantStatus.ts:21-22,45-49,65-79, apps/tenant-service/src/api/tenant.routes.ts:141,168,198

Feature flags: the correct version of the same problem

Two-tier cache (30s in-process, then 5-minute Redis) falling back to a real DB read — global-default row (tenantId IS NULL) overridden by a tenant-specific row when one exists. Updates clear the local cache and publish a Redis pub/sub invalidation so every other service instance drops its own cached copy immediately — the cross-process consistency mechanism the tenant-status cache above is missing.

packages/platform-sdk/src/feature-flags.ts:11-12,21-27,55-61,63-100,113-130

feature_flags / plan_entitlements

feature_flags.tenantId is nullable — NULL row = global default, non-null = a specific tenant's override, unique on (tenantId, flagKey). plan_entitlements is a separate global template table, one row per plan, holding maxUsers/maxBranches (nullable = unlimited), a featureFlags string array, and pricing — explicitly a template copied into a tenant's settings/flags at provisioning time, analogous to how default roles are seeded.

Known Issues & Needs Review

Ranked by real-world impact. Everything below is a confirmed code-reading finding unless marked Needs Review.

Tenant provisioning is non-transactional with no resume mechanism

TenantProvisioner.provision() runs roughly ten sequential steps (tenant row, roles/permissions, workflow templates, admin user + head-office branch, S3 storage, search indexes, feature flags, plan entitlements, welcome email) with no surrounding transaction. A failure partway through (confirmed by a dedicated test simulating an S3 outage) leaves the tenant row, its seeded roles, and its admin user already committed, permanently stuck at PROVISIONING/FAILED, with no automatic rollback or resume — cleanup requires manual database intervention. Explicitly acknowledged in code/test comments as a known, unresolved gap.

Fixed: a "New Tenant" screen now exists

POST /admin/tenants was real, tested, and correctly permission-gated but had zero frontend callers anywhere. A form now exists (apps/web-frontend/src/pages/admin/TenantFormPage.tsx, route admin/tenants/new, permission-gated identically to the tenant list) collecting organization details, plan, and the first Owner account, calling this endpoint directly with no other change to the backend. The frontend schema (schemas/tenant.schema.ts) mirrors the backend's CreateTenantSchema field-for-field, following the same "mirror, don't share" convention already used for every other entity form in this app.

Still open: no way to change a tenant's plan after creation

There is no "change plan" endpoint at all post-provisioning — a plan is fixed at creation time. This was not addressed by the fix above; it's a separate, still-real gap.

Still open: the new tenant's password isn't delivered by the system

The new "New Tenant" form requires the operator to type the first Owner's password directly, same as the API always did — there's still no auto-generated password, no forced-password-change flag, and no credential-bearing email. This still has to be relayed to the client manually, outside the system.

Fixed: the new tenant's ID is now discoverable

Login on this platform requires email, password, and a numeric Tenant ID — but POST /admin/tenants's response (which already included the new tenantId) was previously discarded by the form with no display, and the tenant list showed no ID column either, so a created tenant's ID was genuinely unrecoverable from the UI. The creation flow now shows the new Tenant ID and admin email on a confirmation screen immediately after submit, and the tenant list now has a Tenant ID column for looking one up again later.

Plan entitlement limits are recorded but never enforced

maxUsers/maxBranches are copied into a tenant's settings at provisioning and surfaced read-only on a usage screen with no frontend consumer — but no user-creation or branch-creation code path anywhere checks them. A tenant can exceed its plan's limits freely.

Tenant-suspension status cache has no cross-service invalidation broadcast

Every service caches tenant status in-process for 60 seconds; only the service that performed the suspend/close action invalidates its own cache. Every other running service instance keeps trusting a stale "active" status for up to 60 seconds. Contrast with the feature-flag cache, which correctly uses Redis pub/sub to solve the identical problem — an inconsistent pattern within the same codebase worth aligning.

No server-side guard against chained/nested impersonation

The impersonate endpoint doesn't check whether the calling session is itself an impersonation session before minting a new one. The only protection against stacking impersonation sessions is client-side (the web app's own state management refuses to start a second one) — a direct API call could bypass it. Not confirmed exploited, but the server-side check is genuinely absent.

fifo_valuation feature flag is dead

Seeded as a global default, displayed and toggleable in the admin screen, but never read by any code path — the real FIFO-vs-WACC decision is a per-item field, not this flag. Independently reconfirms a suspicion raised in the Products & Variants audit, now with precise citations.

Two incompatible feature-flag naming conventions coexist unreconciled

Some flags use snake_case (einvoice_enabled, mfa_required), others use dot.case (pos.enabled, sales.loyalty.enabled, hr.tailoring.enabled). The admin screen's hardcoded description list only covers the snake_case set — every dot.case flag actually referenced by real business logic shows a generic "no description" placeholder in the operator UI.

No cross-tenant user management or audit-log viewer for the platform operator

The platform operator's entire cross-tenant surface is tenant lifecycle (suspend/activate/close) plus basic usage figures. There is no way to list, view, or manage a specific customer tenant's user accounts, and the security-audit-log endpoint (the only viewer for impersonation events) is filtered to the caller's own tenant — a platform operator can never see a customer tenant's impersonation activity through this screen.

No single reproducible onboarding script — but the manual sequence is now documented

The UAT seed script assumes tenant IDs 1 and 2 already exist and seeds business data directly via raw SQL — it never calls the actual provisioning endpoint. No migration or script in this repo creates those tenant IDs, and there's still no single command that goes from "nothing" to "a working dev/UAT tenant." What now exists: the real manual sequence — migrate, bootstrap the first PLATFORM_OPERATOR, then provision the tenant via POST /admin/tenants — is written down at Developer Platform → Environment Setup & Bootstrapping, including who is actually trusted to run the bootstrap scripts and why.

Confirmed solid: tenant CRUD permission-gating

Unlike nearly every other module audited so far, tenant lifecycle management itself has no role/permission mismatch. This wasn't always true — a documented prior vulnerability (zero permission check at all) has been fixed and is locked in with a dedicated regression-test suite covering unauthenticated, under-permissioned, and correctly-permissioned cases.

Confirmed solid: impersonation design, and a previously-flagged concern that turned out to be already resolved

Impersonation is properly scoped to a single tenant, time-limited to one hour, and fully audit-logged on both start and end, using the target user's own permissions rather than the impersonator's. Separately, a concern carried over from an earlier session — that several distributed-systems admin consoles (dead-letter queue, saga, schema registry, event store, projections) were gated on an overly broad audit-log permission mismatched against granular frontend checks — was checked directly against the current code and found to already be fixed: all five now consistently use their own specific permission, matching the frontend exactly.

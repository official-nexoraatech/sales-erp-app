Purchase Overview

Fourth module documented from a direct code audit. Covers Purchase Orders, Goods Receipt (GRN), Purchase Returns/Debit Notes, and Supplier Payments & Expenses.
Two core screens in this module cannot currently be used

Creating a Goods Receipt Note (GRN) and creating a Purchase Return both fail every time you submit them, due to confirmed defects in the forms themselves — not something you're doing wrong. Between them, that means the "receive what you ordered" step of purchasing doesn't work end-to-end today. Full detail on both is on their own pages and on Known Issues. Treat this module as PO-creation-and-approval-only until these are fixed.

What's here

  • Purchase Orders — request goods from a supplier; single-level approval with a credit-limit check.
  • Goods Receipt (GRN) — record what actually arrived against an approved PO. Currently broken — see below.
  • Purchase Returns & Debit Notes — send goods back to a supplier. Currently broken — see below.
  • Supplier Payments & Expenses — pay suppliers and log purchase-related expenses.

Supplier records themselves (create/edit a supplier) are managed under a separate Suppliers section, not inside this module.

Creating a Purchase Order

  1. New Purchase Order

    Purchase → Orders → New. Pick a supplier and add line items with quantity and rate.

  2. Submit

    Moves the PO from Draft to Submitted, ready for approval.

  3. Approve

    A single approval step finalizes it — there's no multi-level sign-off chain.
    Credit-limit check on approval

    If the supplier has a credit limit configured, approving a PO that would push their outstanding balance over that limit is blocked unless the approver has override authority.

Once approved, the PO becomes eligible for receiving against — see Receiving Goods.

Cancelling an approved or partially-received PO

The Cancel option is only shown in the list for Draft/Submitted purchase orders. An Approved or Partially Received PO can technically still be cancelled through the system, but there's no button for it in the current screen — treat those as not cancellable until this is clarified or added.

Printing a PO

The "Download PDF" action doesn't currently produce a file — no PDF is generated for purchase orders yet.

Receiving Goods (GRN)

Creating a GRN currently fails every time

This is a confirmed defect: the "New GRN" form doesn't send the exact fields the server expects, so every submission is rejected. There is no workaround from this screen today — it needs a development fix. If goods have physically arrived and you need stock updated urgently, that has to happen as a manual stock adjustment in Inventory in the meantime, not through this screen.

How this is meant to work

Documenting the intended flow so it's ready the moment it's fixed:

1
Open an Approved (or Partially Received) PO, click Receive
→
2
Adjust received quantity per line — partial receipt is fully supported by design
→
3
Approve the GRN — stock and cost update immediately

A GRN can only ever be created against an approved PO — there's no way to log received goods without one, by design.

Price variance flag

If the rate you're receiving at differs from the PO rate by more than 5%, the GRN is routed for a second approval instead of auto-approving — a safeguard against silent overbilling, sometimes described internally as a "3-way match" (it's really a PO-vs-received-goods check; see Numbering & 3-Way Match for the precise scope).

Purchase Returns & Debit Notes

Creating a Purchase Return currently fails every time

Confirmed defect: the return form doesn't let you pick which items are being returned, and its reason dropdown offers choices the server doesn't recognize. Every submission is rejected. No workaround exists from this screen today.

How this is meant to work

A return is created against an already-approved GRN — you're sending back goods you already received. Approving it restores the paperwork trail and automatically issues a Debit Note against the supplier, reducing what you owe them.

Debit note value doesn't reduce your ledger cost automatically

Even once return creation is fixed: approving a purchase return correctly reduces stock quantity, but does not adjust the ₹ cost value your system holds for that item. Over time, if you use purchase returns often, your recorded stock value can drift from its true value. Flag this to your accountant if purchase returns are common in your business.

Supplier Payments & Expenses

Paying a supplier

Record a payment mode, amount, and (for cheque) the details — the same shape as customer payments in Sales & Invoicing. Payments can be allocated against specific GRNs, and a bounced cheque correctly reverses the payment.

Supplier statement / outstanding balance isn't wired up

The system has a real, working calculation of what you owe a supplier — but there's currently no screen that shows it to you, and even where the API for it exists, most roles don't have permission to call it. Don't rely on an in-app "supplier statement" yet; check with your accountant for the true outstanding balance.

Expenses

Log a purchase-related expense, submit it, and have it approved and marked paid.

Branch is not selectable when creating a return, payment, or expense

These three forms currently record everything against a single fixed branch regardless of which branch you're actually working from. If you operate more than one branch, check with your admin about how this affects your branch-level reporting until it's fixed.

Purchase Manager can't check a supplier's outstanding balance

The permission that gates both supplier-statement endpoints is never included in the Purchase Manager role's default permission set — only Owner/Admin have it. Since checking a supplier's balance before approving a PO is exactly the kind of thing a Purchase Manager would want to do, this looks like a role-configuration gap worth fixing, not an intentional restriction.

Supplier records are managed by a different service than Purchase

Supplier CRUD (create/edit/delete a supplier) is technically implemented in the same backend service as Sales & Invoicing's customer records, not Purchase's own backend. This is invisible to end users — the Suppliers screen works the same either way — but matters for anyone debugging supplier-related issues.

Numbering & 3-Way Match

PO and GRN numbers are typed manually, not generated

Same pattern already flagged for Sales & Invoicing: the approver types the PO/GRN number themselves, with only a per-tenant uniqueness check — no sequence enforcement. Purchase-return and debit-note numbers are auto-generated, but from a timestamp, not a true gapless counter. If sequential document numbering matters for your compliance/audit needs, none of these four document types currently guarantee it.

"3-way match" is a bit of a misnomer here

Internally, a price-variance check between the Purchase Order and the received goods is sometimes called a "3-way match." In practice it only compares two documents — the PO and what was actually received — and flags anything more than 5% off rate for a second approval. The supplier's own invoice is recorded as reference information on the GRN but isn't independently entered or amount-matched against it. If your accounting process expects a true PO-vs-goods-receipt-vs-supplier-invoice match, that third leg doesn't exist today.

Limitations to Know

Full technical detail is on Known Issues.

  1. GRN creation is broken end-to-end — the core "receive what you ordered" step of purchasing doesn't work today. Highest-priority fix if purchasing is active for this tenant.
  2. Purchase Return creation is broken end-to-end — same severity as #1.
  3. Purchase returns don't adjust stock value — quantity is correct, ₹ cost isn't, and will drift for tenants using returns regularly.
  4. Purchase Manager can't view supplier outstanding balances — a role-configuration gap.
  5. Purchase returns, supplier payments, and expenses are all recorded against a single fixed branch regardless of the user's actual branch.
  6. PO PDF generation and the Landed Cost UI don't exist — both are stubs/API-only today.

supplierApi in the frontend routes every call through the 'sales' gateway key; the implementation is apps/sales-service/src/api/supplier.routes.ts. purchase-service only reads the shared suppliers table (credit-limit checks, RCM checks, balance projections) — it doesn't own the CRUD, despite Purchase Manager holding SUPPLIER_CREATE/SUPPLIER_EDIT. apps/web-frontend/src/api/endpoints.ts:164-178

PO → GRN is a separate linked document, not a "convert"

Unlike Sales' quotation→invoice conversion, a GRN is its own record referencing purchaseOrderId by FK. The list page's "Receive" action navigates to /purchase/grns/new?poId=<id>, which loads the PO and pre-fills a receiving grid. Partial receipt is a first-class concept: each PO line tracks orderedQty vs receivedQty independently. apps/web-frontend/src/pages/purchase/PurchaseOrdersPage.tsx:141-143, GRNCreatePage.tsx:58-77

GRN creation — the exact field mismatch

Frontend and backend disagree on the payload shape

GRNCreatePage.tsx submits receivedDate and never includes branchId. The backend's CreateGRNSchema requires a field named grnDate (not receivedDate) and a required branchId. Every submission fails Zod validation. Separately, GRNsPage.tsx reads/renders r.receivedDate for its "Received" column, but the actual column is grnDate — so even a GRN created directly via the API would show an empty/invalid date in this list. apps/web-frontend/src/pages/purchase/GRNCreatePage.tsx:117-127, apps/purchase-service/src/api/grn.routes.ts:25-35, GRNsPage.tsx:22-31,101

Purchase Return creation — the exact defects

Empty line items plus an invalid reason enum

PurchaseReturnsPage.tsx always submits lines: [] with no item picker in the UI at all — the backend requires at least one line. Its reason dropdown also offers DEFECTIVE/EXCESS_QTY, values the backend's enum doesn't recognize (it expects QUALITY_ISSUE/WRONG_ITEM/EXCESS_QUANTITY/DAMAGED/OTHER) — the default selected value alone would fail validation even with lines fixed. Separately, the Debit Notes tab reads r.total for its Amount column, but the actual field is amount — renders as ₹NaN. PurchaseReturnsPage.tsx:51,62,218-226,35,132, apps/purchase-service/src/api/purchase-return.routes.ts:20-29

Data Model

packages/db-client/src/schema/purchase.ts. Supplier master data lives in packages/db-client/src/schema/master.ts (owned/CRUD'd by sales-service).

purchase_orders / purchase_order_lines

Status: DRAFT|SUBMITTED|PENDING_APPROVAL|APPROVED|PARTIALLY_RECEIVED|RECEIVED|CLOSED|CANCELLED. CLOSED is declared but never assigned by any code path — dead enum value. Also: purchase_order_history, purchase_order_amendments.

packages/db-client/src/schema/purchase.ts:16-135

grns / grn_lines

Status: DRAFT|PENDING_APPROVAL|APPROVED|REJECTED. Carries supplierInvoiceNumber/supplierInvoiceDate directly — there is no separate supplier-bill/purchase-invoice table; the supplier's bill reference is folded into the GRN record itself. Also: grn_history, landed_costs (cost types customs/freight/insurance/handling/other; allocation by value/quantity/weight — computed but never fed into item valuation).

packages/db-client/src/schema/purchase.ts:138-264

purchase_returns / purchase_return_lines / debit_notes

Return status: DRAFT|PENDING_APPROVAL|APPROVED|CANCELLED; reason: QUALITY_ISSUE|WRONG_ITEM|EXCESS_QUANTITY|DAMAGED|OTHER. Debit note status: OPEN|PARTIALLY_APPLIED|APPLIED|CANCELLED.

packages/db-client/src/schema/purchase.ts:326-415

supplier_payments / supplier_payment_allocations / expenses

Payment status: PAID|PARTIALLY_ALLOCATED|FULLY_ALLOCATED|BOUNCED|CANCELLED, includes post-dated-cheque fields. Expense status: DRAFT|SUBMITTED|PENDING_APPROVAL|APPROVED|PAID. projection_supplier_balance is the CQRS read model, updated inline inside GRN-approve/return-approve/payment-create transactions.

packages/db-client/src/schema/purchase.ts:267-498

Numbering

poNumber/grnNumber are free-text, typed manually at approval time, unique per tenant only. returnNumber/debitNoteNumber auto-generate as PR-{tenantId}-{Date.now()}/DN-{tenantId}-{Date.now()} — timestamp-based, same pattern as invoice numbering already flagged in Sales & Invoicing. No sequence-generation service exists anywhere in the repo.

Known Issues & Needs Review

Ranked by real-world impact. Everything below is a confirmed code-reading finding.

GRN creation is broken end-to-end

GRNCreatePage.tsx submits receivedDate and omits branchId; the backend's CreateGRNSchema requires grnDate and a required branchId. Every submission fails validation. GRNsPage.tsx also reads the wrong field name for its date column. GRNCreatePage.tsx:117-127, grn.routes.ts:25-35, GRNsPage.tsx:22-31,101

Purchase Return creation is broken end-to-end

PurchaseReturnsPage.tsx always submits lines: [] (no item picker exists) against a schema requiring at least one line, and its reason dropdown includes two values (DEFECTIVE, EXCESS_QTY) the backend enum doesn't accept. The Debit Notes tab separately reads a non-existent total field instead of amount. PurchaseReturnsPage.tsx:51,62,218-226,35,132, purchase-return.routes.ts:20-29

Purchase returns don't update stock valuation

PurchaseReturnService.approve() correctly deducts quantity and writes a STOCK_OUT ledger row, but never imports or calls ValuationService — WACC/FIFO cost data isn't adjusted. Confirmed unresolved from an earlier phase's own documented gap.

Purchase Manager lacks SUPPLIER_STATEMENT_VIEW

Both supplier-statement/outstanding-balance endpoints require this permission, which is absent from the Purchase Manager role's default grants — only Owner/Admin have it via their all-permissions wildcard. apps/tenant-service/src/rbac/role-defaults.ts:76-99

Hardcoded branchId: 1

PurchaseReturnsPage.tsx, SupplierPaymentsPage.tsx, and ExpensesPage.tsx all hardcode branchId: 1 in their create payloads rather than using the user's actual branch — every purchase return, supplier payment, and expense is attributed to branch 1 regardless of who creates it.

Landed costs have zero UI

landedCosts/addLandedCost/allocateLandedCost client functions exist but are called from nowhere in the frontend. Separately, even when allocated via direct API use, landed costs never feed into item WACC/FIFO cost — GRNService.approve() reads the plain grnRate, not the landed-cost-adjusted effectiveUnitCost.

"3-way match" is really a 2-document price-variance check

Only PO rate vs. received rate is compared (5% threshold); the supplier's own invoice is stored as reference metadata on the GRN, never independently amount-matched.

Two competing, both-unused supplier statement implementations

Sales-service's version is a stub that returns empty arrays; purchase-service's version is a real implementation joining the supplier-balance projection with recent GRNs/payments. Neither is called from any frontend page.

Dead permission/status surface

purchase_orders.status's CLOSED value is never assigned by any code path. PAYMENT_OUT_CANCEL/PAYMENT_OUT_APPROVE permission constants gate no route. GRN_UPDATE is granted to Purchase Manager but only gates GRN attachment uploads — there is no GRN edit endpoint. PO_AMEND has a full backend implementation but no frontend caller anywhere.

PO PDF generation is a stub

GET /purchase-orders/:id/pdf only returns the pdfUrl column; nothing in purchase-service ever writes to it.

Test coverage gap

PO approval/credit-limit/amend, GRN over-receipt guards, RCM detection, and purchase-return ledger writes are well tested. No test exercises valuation behavior on GRN approval or purchase-return approval, LandedCostService, or ExpenseService — consistent with the still-open valuation-drift gap having no regression guard. No TODO/FIXME comments exist anywhere in purchase-service.

Confirmed solid: PO approval and over-receipt guards

Supplier credit-limit checking on PO approval, and the double-guarded (optimistic + atomic) over-receipt protection on GRN creation/approval, are both real, race-safe, and tested — including a specific concurrent-approval race guard.
